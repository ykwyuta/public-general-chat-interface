# Google認証 テストケース

本ドキュメントは、Google認証機能（`docs/google-auth-design.md`）に関するPlaywrightでのE2Eテスト作成に向けたテストケースをまとめたものです。

## 1. ログイン画面の表示制御（環境変数による切り替え）
- **テストケース 1.1: Google認証が無効な場合の表示**
  - **事前条件**: サーバーの環境変数（`.env.local`等）に `GOOGLE_CLIENT_ID` が設定されていない。
  - **手順**: ログイン画面にアクセスする。
  - **期待される結果**: 従来のデモ認証（ユーザー名・パスワード）のフォームのみが表示され、「Googleでログイン」ボタンは表示されないこと。
- **テストケース 1.2: Google認証が有効な場合の表示**
  - **事前条件**: サーバーの環境変数に `GOOGLE_CLIENT_ID` が設定されている。
  - **手順**: ログイン画面にアクセスする。
  - **期待される結果**: デモ認証のフォームに加えて、「Googleでログイン」ボタンが表示されること。

## 2. 初回ログインとユーザー名設定フロー
- **テストケース 2.1: 初回ログイン時の画面遷移と初期値の自動生成**
  - **事前条件**: データベースに登録されていない新規のGoogleアカウント（例: `test.user@example.com`）でログイン処理を行う。
  - **手順**:
    1. 「Googleでログイン」ボタンから認証を完了する。
    2. 認証後、アプリケーションにリダイレクトされる。
  - **期待される結果**: ユーザー名設定画面（UsernameSetupPage）が表示されること。メールアドレスのローカルパートをもとにサニタイズされた初期値（例: `test_user`）が入力欄にセットされていること。
- **テストケース 2.2: ユーザー名の重複回避（自動連番の提案）**
  - **事前条件**: データベースに `test_user` というユーザー名が既に存在する。
  - **手順**: 別の未登録アカウント（例: `test.user@otherdomain.com`）で初回ログインする。
  - **期待される結果**: ユーザー名設定画面の初期値として、既存のものと重複しないよう連番が付与された値（例: `test_user2`）が自動提案されること。
- **テストケース 2.3: ユーザー名のバリデーションエラー**
  - **事前条件**: 初回ログイン後のユーザー名設定画面が表示されている。
  - **手順**: 以下の不正なユーザー名を入力し、確定を試みる。
    1. 短すぎる文字列（例: `a`）
    2. 長すぎる文字列（31文字以上）
    3. 許可されていない文字を含む（例: `User-Name`, `user@name`）
  - **期待される結果**: クライアント側（またはサーバー側）のバリデーションエラーメッセージが表示され、登録が進行しないこと。（※制約: 2〜30文字、半角英小文字・数字・アンダースコアのみ）
- **テストケース 2.4: ユーザー名の確定とメイン画面への遷移**
  - **事前条件**: ユーザー名設定画面で有効かつ重複しないユーザー名が入力されている。
  - **手順**: 「確定」ボタンを押下する。
  - **期待される結果**: 登録が成功し、メインのチャット画面（AppLayout）へ遷移して認証済み（authenticated）状態となること。

## 3. 2回目以降のログインフロー
- **テストケース 3.1: 登録済みアカウントでのログイン（スキップ処理）**
  - **事前条件**: 既にユーザー名設定を完了したGoogleアカウントが存在する。
  - **手順**: ログアウト状態から該当アカウントで「Googleでログイン」を行う。
  - **期待される結果**: ユーザー名設定画面が表示されることなく、直接メインのチャット画面へ遷移すること。

## 4. ドメイン制限機能
- **テストケース 4.1: 許可されたドメインでのログイン**
  - **事前条件**: 環境変数 `GOOGLE_ALLOWED_DOMAINS` に `example.com` が設定されている。
  - **手順**: `test@example.com` のアカウントでログインを試みる。
  - **期待される結果**: 正常にログインフロー（初回または2回目以降）へ進むこと。
- **テストケース 4.2: 許可されていないドメインでのアクセス拒否**
  - **事前条件**: 環境変数 `GOOGLE_ALLOWED_DOMAINS` に `example.com` が設定されている。
  - **手順**: `test@gmail.com` など対象外のドメインでログインを試みる。
  - **期待される結果**: 認証エラー画面（`/auth/error`）へ遷移し、「このアカウントでのアクセスは許可されていません」といったエラーメッセージが表示され、システム内にログインできないこと。

## 5. ログアウト処理
- **テストケース 5.1: Google認証からの完全なログアウト**
  - **事前条件**: Googleアカウントでログインし、メイン画面が表示されている。
  - **手順**:
    1. 画面上の「ログアウト」ボタンをクリックする。
    2. ログイン画面へ遷移後、ブラウザをリロードする。
  - **期待される結果**:
    1. 状態管理（Zustand等）のデータがクリアされ、ログイン画面へ戻ること。
    2. リロードしてもメイン画面へ自動遷移せず、セッションが完全に破棄されていること。
