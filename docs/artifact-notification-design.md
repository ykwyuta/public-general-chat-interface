# アーティファクト共有・通知機能 設計書

## 1. 概要・目的

### 背景

現在のシステムでは、チャットで生成されたアーティファクト（コード、ドキュメントなど）をユーザー間で直接共有したり、それに基づいて他のユーザーに作業を依頼したりする機能が不足している。タスク機能は複数の人間とLLMが協力する機能であるが、単発の依頼や成果物の共有には重量すぎる場合がある。

### 目的

チャットのメッセージ入力欄において、特定のユーザーへメンション（`@username`）を付け、同時にアーティファクト（`#artifact_name`）を添付してメッセージを送信できる機能を追加する。
これにより、メンションされたユーザーは通知を受け取り、その通知からアーティファクトとメッセージを引き継いだ新しいチャット（依頼チャット）をシームレスに開始できるようになる。

### 主要機能

1. **メンションとアーティファクト指定**: メッセージ入力時に `@` でユーザー、`#` でアーティファクトを補完・指定できる。
2. **通知機能**: メンションされたユーザーにアプリ内通知が届く。
3. **依頼チャットの開始**: 受信者は通知一覧から、送られたメッセージとアーティファクトを引き継いで新しいチャットを開始できる。
4. **依頼情報の表示**: 依頼チャットの画面上部に「依頼事項」「依頼者」「依頼日時」がバナーとして表示される。

---

## 2. ユースケース・画面遷移フロー

### フロー1: 依頼の送信
1. ユーザーAがチャット画面でメッセージを入力。
2. `@` を入力すると、システム内のユーザー一覧がオートコンプリートで表示される。ユーザーB（例: `@userB`）を選択。
3. `#` を入力すると、現在のチャットセッション内のアーティファクト一覧がオートコンプリートで表示される。アーティファクト（例: `#index.html`）を選択。
4. メッセージ（例: `@userB #index.html このデザインのレビューをお願いします`）を送信。
5. サーバーがメンションを検知し、ユーザーBに対する通知を作成する。

### フロー2: 通知の受信と依頼チャットの開始
1. ユーザーBがシステムにログイン（または使用中）すると、ヘッダーの「通知ベル」アイコンにバッジが表示される。
2. 通知アイコンをクリックすると、通知パネルがドロップダウンまたはサイドパネルで表示される。
3. 通知一覧からユーザーAからの依頼通知（メッセージのプレビュー付き）をクリックする。
4. ユーザーBの画面が新しいチャット画面に遷移する。
   - この時、共有されたアーティファクトがパネルに初期状態としてセットされる。
   - チャットのヘッダー直下に、依頼情報（依頼事項: "このデザインのレビューをお願いします", 依頼者: "ユーザーA", 依頼日時: "YYYY-MM-DD HH:mm"）が固定表示される。

## 3. データモデル・スキーマ変更

### 3.1 `notifications` テーブルの新規追加

ユーザー宛ての通知を保存する新しいテーブル。

```sql
CREATE TABLE IF NOT EXISTS notifications (
  id              TEXT PRIMARY KEY,
  user_username   TEXT NOT NULL, -- 受信者のusername
  sender_username TEXT NOT NULL, -- 送信者のusername
  message         TEXT NOT NULL, -- メッセージ本文（依頼内容）
  artifact_id     TEXT,          -- 共有されたアーティファクトのID（必要に応じてJSON配列等に変更可能）
  source_conv_id  TEXT,          -- 元の会話ID
  is_read         INTEGER NOT NULL DEFAULT 0, -- 既読フラグ (0: 未読, 1: 既読)
  created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_notifications_user
  ON notifications(user_username, created_at DESC);
```

### 3.2 `conversations` テーブルの拡張

依頼として開始されたチャットであることを識別するため、`conversations` テーブルに依頼情報を保存するカラムを追加する。

```sql
ALTER TABLE conversations ADD COLUMN request_message TEXT;      -- 依頼事項
ALTER TABLE conversations ADD COLUMN request_sender TEXT;       -- 依頼者のusername
ALTER TABLE conversations ADD COLUMN request_created_at TEXT;   -- 依頼日時
```
*(SQLiteでの `ALTER TABLE` 制約により、スキーママイグレーションまたは新規テーブル作成時の定義変更が必要となる場合があります)*

## 4. API設計

### 4.1 通知関連 API

#### `GET /api/notifications`
- **概要**: ログイン中ユーザーの通知一覧を取得する。
- **レスポンス**:
  ```json
  [
    {
      "id": "notif_1",
      "sender_username": "userA",
      "message": "このデザインのレビューをお願いします",
      "artifact": { "id": "art_1", "filename": "index.html", ... },
      "is_read": false,
      "created_at": "2023-10-27T10:00:00Z"
    }
  ]
  ```

#### `POST /api/notifications/[id]/read`
- **概要**: 通知を既読状態にする。
- **レスポンス**: `200 OK`

### 4.2 依頼チャット作成 API

#### `POST /api/conversations/from-request`
- **概要**: 通知を元に新しい依頼チャットを作成する。
- **リクエスト**:
  ```json
  {
    "notification_id": "notif_1"
  }
  ```
- **レスポンス**:
  ```json
  {
    "conversation_id": "conv_2",
    "redirect_url": "/chat/conv_2"
  }
  ```
- **内部処理**:
  1. 通知IDから情報（メッセージ、アーティファクト、送信者）を取得。
  2. 新しい `conversation` レコードを作成し、`request_message`, `request_sender`, `request_created_at` を設定する。
  3. 初期のアーティファクト情報を初期メッセージとして（あるいは状態として）設定する。
  4. 通知を既読にする。

### 4.3 メッセージ送信 API のフック
- 既存のチャットメッセージ送信API（例: `/api/chat/route.ts` またはそれに類するエンドポイント）において、メッセージ内容を解析（メンション `@` とアーティファクト `#` の抽出）する処理をサーバー側に追加する。
- 宛先ユーザーが存在し、アーティファクトが指定されている場合、`notifications` テーブルにレコードを挿入する。

## 5. フロントエンド改修内容

### 5.1 入力欄のオートコンプリート拡張
- **対象ファイル**: `src/components/chat/ChatInput.tsx` (またはそれに相当する入力コンポーネント)
- **改修内容**:
  - 既存の `@` メンション（タスク機能で実装済み/予定のもの）に加え、通常のチャットでもユーザー一覧をサジェストできるようにする。
  - `#` が入力された際に、Zustandのストア（例: `chatStore.ts`）から現在の会話の `artifacts` リストを取得し、サジェスト表示するポップアップを実装する。

### 5.2 通知一覧パネルの追加
- **対象ファイル**: `src/components/layout/Header.tsx`, `src/components/notification/NotificationPanel.tsx` (新規)
- **改修内容**:
  - ヘッダーにベル型アイコンを配置。未読件数をバッジ表示。
  - クリックでドロップダウンまたはサイドパネルを開き、`GET /api/notifications` の結果をリスト表示。
  - リスト項目をクリックすると、`POST /api/conversations/from-request` を呼び出し、結果のURLへ遷移する。

### 5.3 依頼事項バナーの表示
- **対象ファイル**: `src/components/chat/ChatRoom.tsx` (またはメインのチャット画面コンポーネント)
- **改修内容**:
  - 現在の `conversation` データに `request_message`, `request_sender` 等が含まれている場合、チャット画面上部（ヘッダーとメッセージリストの間）に専用のバナーコンポーネント（`RequestBanner.tsx`）を表示する。
  - バナーには「依頼者」「依頼日時」「依頼事項」を明確に表示する。

### 5.4 Zustandストアの拡張
- **対象ファイル**: `src/stores/notificationStore.ts` (新規作成を推奨)
- **改修内容**:
  - 未読通知数、通知一覧のフェッチ、既読化のアクションを管理する。

## 7. 備考・留意事項

- **アーティファクトの所有権とアクセス制御**: 元のチャットに属するアーティファクトのデータを、依頼先のチャットでも参照またはコピーできるようにする必要がある。アーティファクトの実体がコピーされるか、参照が渡されるかの設計（JSONのディープコピー等）を実装時に確定させる。
- **タスク機能との住み分け**: この機能は「単発の依頼と共有」に特化しており、タスク機能（目的と完了条件を持つ）とは目的が異なる。ユーザーが適切に使い分けられるよう、UI上での導線を工夫する。
